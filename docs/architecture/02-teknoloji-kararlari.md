# 02 — Teknoloji Kararları

> Eksen: **tasarım / teknik**. Bu dosya **kararların kaydıdır** — yerleşik kabul edilir;
> kullanıcı açıkça istemedikçe yeniden açma. Karar değişirse aşağıdaki "Karar kaydı"na tarihli satır ekle.

## Sabit kararlar

| Konu | Karar |
| --- | --- |

| Queue | BullMQ + Redis |
| Realtime | Socket.IO + Redis adapter; DB/outbox source of truth |
| Push | Expo Notifications |
| Drag-drop | Atlassian Pragmatic Drag and Drop (yalnızca web) |
| Auth | Better Auth |
| Web UI sistemi | shadcn/ui (yalnızca); başka web component library yok |

- **2026-05-13** — Faz 5 ([DEM-28](https://linear.app/demirkol/issue/DEM-28)) "önce belge" kararları (`faz-baslat 5`): (a) **Realtime event yayın stratejisi = worker queue + `realtime_events` outbox tablosu** (direct in-process emit DEĞİL). Akış: mutation tx içinde `realtime_events` INSERT (atomic) → tx COMMIT → `void enqueue({ eventId })` (best-effort) → `apps/worker` `pusula-realtime-publish` job alır → Socket.IO server'a Redis pub/sub üzerinden publish → `published_at` damgalanır. Periyodik sweeper (60s aralıklı) `published_at IS NULL AND created_at < NOW() - 30s` satırları yeniden enqueue eder — replay garantisi. Gerekçe: kullanıcı kararı (event kaybı sıfır tolerans); Faz 6 `notification_outbox` ile birebir simetrik pattern; audit trail (`realtime_events` 7 gün retention); Redis blip / API restart dayanıklılığı. Latency trade-off kabul edildi (~200-500ms vs direct emit ~30ms; `seq` + refetch reconciliation latency hissini telafi eder). (b) **Redis adapter zorunlu** — Faz 5 başlangıcında tek `apps/api` instance bile olsa adapter kurulur (worker → API publish için Redis pub/sub kanalı zaten lazım + multi-instance scale-out'a hazırlık). Sticky session ihtiyacı yok (WebSocket-only transport; long-polling fallback Faz 5'te kapalı). (c) **`viewer` rolü `board:{boardId}` socket room'una join eder** — event'leri görür, optimistic mutation atmaz (mevcut Faz 4C draggable mount disiplini korunur). Arşivli board → join reddi. (d) **Echo ayıklama mekaniği** — `useOptimisticBoardMutation` (Faz 4C) in-flight `clientMutationId` set'i ile entegre; event listener `inFlightSet.has(event.clientMutationId)` ise skip eder. Gerekçe: kullanıcı kendi optimistic update'ini echo event'le ikinci kez uygulamaz. (e) **`seq` gap → `board.get` refetch** — `boards.version` aynası `lastAppliedSeq`; gap'te outbox replay client'a değil, server `board.get` cevabı authority. Reconnect resync de aynı pattern (tek refetch). (f) **Kapsam dışı kararları:** comment/checklist/label/member realtime → Faz 6 (`notification_outbox` ile birlikte), presence (kim board'da/imleç) → Faz 6/7 (özel taşıma, outbox'a yazmaz — in-memory), mobile realtime → Faz 7. Detay → `05-board-mekanigi.md` §5.3, `03-backend.md` "Faz 5 — Socket.IO server", `06-bildirim-altyapisi.md` "Realtime event yayın katmanı (Faz 5)", `08-web-ve-mobil.md` §8.1.10, `04-veri-katmani.md` (`realtime_events` tablosu Faz 5 kapsamı), `10-platform.md` "Realtime altyapısı (Faz 5)".
- **2026-05-13** — Faz 4 ([DEM-27](https://linear.app/demirkol/issue/DEM-27)) "önce belge" kararları (`faz-baslat 4`): (a) **`clientMutationId` = UUID v4** (`crypto.randomUUID()` client tarafında üretilir, opsiyonel input alanı `z.string().uuid().optional()` — gönderim opsiyonel, ama Faz 4C UI tüm collaborative mutation'larda mutlaka üretir). (b) **Faz 4'te scope = board/list/card collaborative mutation'ları** — `card.move` / `moveToList` / `copy` / `create` / `update` / `archive` / `complete` / `uncomplete`, `list.move` / `create` / `update` / `archive`, `board.create` / `update` / `archive`. Comment/checklist/label/member optimistic UI Faz 5/6'ya bırakılır (realtime echo + activity feed ile birlikte). Gerekçe: bu alanlar Faz 5 (realtime — [DEM-28](https://linear.app/demirkol/issue/DEM-28)) sırasında zaten dokunulacak; iki kez yeniden yazmamak için orada optimistic'lenir. (c) **Faz 4'te yalnız idempotency log iskeleti** — `packages/api` prosedürlerinde `clientMutationId` parse edilir, `ctx.clientMutationId` olarak gövdeye taşınır, log/audit'e yazılır, activity event'lere alan eklenir. **Short-window dedupe (kısa pencerede aynı `clientMutationId` ile gelen retry'ı server tarafında düşürme) Faz 5'e bırakılır** — outbox + realtime echo ile birlikte tek yerden kurulur. Gerekçe: Faz 4'ün hedefi optimistic UI; gerçek dedupe Faz 5'in ilgi alanı. (d) **Web cache modeli `apps/web/src/lib/board-cache/` modülünde toplanır** — `keys.ts` (query key factory) + `types.ts` (BoardCache/ListCache/CardCache, tRPC output'undan türetilmiş) + `primitives.ts` (pure cache update fonksiyonları) + `mutations.ts` (`useOptimisticBoardMutation` higher-order hook). Component'lar literal query key yazmaz, cache erişimi tek modülden geçer. Gerekçe: Faz 3B'de cache primitives `board-dnd-cache.ts` özelinde kuruldu; Faz 4'te tüm board/list/card mutation'larına genişletildiğinde ortak modül zorunlu. Detay → `05-board-mekanigi.md` §5.2, `08-web-ve-mobil.md` §8.1.9, `process/02-mvp-faz-plani.md` (Faz 4 alt işleri).
